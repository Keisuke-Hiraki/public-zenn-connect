## 今日のゴール

CLAUDE.md(Claude Codeに常時読み込ませる指示書)の仕組みと書き方を理解し、自分の業務に合わせたCLAUDE.mdを実際に作ってみる。

## 座学

### CLAUDE.mdとは何か

これまでの2回で、Claude Codeは「ハーネス」として動作し、対話のたびに指示を出せることを体験しました。しかし、毎回同じ前提や好みを説明し直すのは非効率です。そこで使うのが**CLAUDE.md**です。

CLAUDE.mdは、作業フォルダに置いておくだけで、Claude Codeが会話を始めるたびに自動的に読み込む指示書です。「私はこういう業務をしていて、こういう形式でアウトプットしてほしい」といった前提を一度書いておけば、以後のやり取りで毎回説明し直す必要がなくなります。いわば、Claude Codeに渡す"自己紹介と業務マニュアル"です。

### /init コマンドで最初のCLAUDE.mdを作る

CLAUDE.mdは手書きで一から作ることもできますが、`/init` というスラッシュコマンドを使うと、Claude Code自身に作業フォルダの中身を調べさせて、たたき台となるCLAUDE.mdを自動生成させることができます。

```
/init
```

コード中心のプロジェクトであれば、このコマンドはディレクトリ構成やビルド方法を調べてCLAUDE.mdの下書きを作ります。TAM(テクニカルアカウントマネージャー)のような非エンジニア業務のフォルダでは、フォルダの中にある報告書のひな形やメモを踏まえた下書きが作られます。あくまで下書きなので、内容を見て自分の言葉に調整していくのが基本的な使い方です。

### CLAUDE.mdの置き場所とスコープ

CLAUDE.mdは置く場所によって、どこまでの範囲に効果が及ぶかが変わります。これを「スコープ」と呼びます。スコープには4段階があり、複数が同時に有効な場合は、後から読み込まれるものほど優先されます。

| スコープ | 保存場所(例) | 効果が及ぶ範囲 |
|---|---|---|
| Managed(組織全体) | 会社が配布する共通設定 | 全社員・全プロジェクト共通のルール |
| User(個人全体) | `~/.claude/CLAUDE.md` | 自分が使うすべてのプロジェクトに共通する個人設定 |
| Project(プロジェクト) | 作業フォルダ直下の `CLAUDE.md` | そのプロジェクトに関わるメンバー全員で共有するルール |
| Local(個人・当該プロジェクトのみ) | 作業フォルダ直下の `CLAUDE.local.md` | 自分だけのメモ(社外秘の顧客情報など、Gitで共有したくない内容) |

優先順位は「Managed → User → Project → Local」の順に読み込まれ、後から読み込まれたLocalの内容が最も優先されます。個人の業務フォルダで使う場合は、まずProjectスコープ(フォルダ直下のCLAUDE.md)から始めるのがわかりやすいでしょう。

### 他のファイルを読み込む(@import)

CLAUDE.mdの中から、`@ファイル名` という書き方で別のファイルの内容を読み込ませることができます。これを使うと、長くなりがちな指示を用途ごとに分割して管理できます。

```markdown
# プロジェクト概要
詳しいセットアップ手順は @docs/setup.md を参照してください。

# 個人の作業ルール
@~/.claude/my-notes.md
```

一つのCLAUDE.mdにすべてを詰め込むのではなく、「共通の内容はCLAUDE.md本体に、個人的なメモは別ファイルに分けて@importする」といった使い方ができます。

### Auto Memory(自動メモリ)機能

CLAUDE.mdが「人間が書く指示書」であるのに対し、**Auto Memory**はClaude Code自身が会話の中で得た気づきを自動的に記録していく仕組みです。例えば「このフォルダの報告書は特定の言い回しを避けてほしい」と一度伝えると、その内容が自動的にメモとして保存され、次回以降の会話でも参照されるようになります。

Auto Memoryは既定で有効になっており、`/memory` コマンドで保存された内容を確認・編集・削除できます。CLAUDE.mdが「意図的に書くルール」であるのに対し、Auto Memoryは「Claude Codeが学習した経験則」と捉えると違いがわかりやすくなります。

## ミニハンズオン

### 手順1: TAM業務向けのCLAUDE.mdを書いてみる

第1回で作った作業フォルダ(`~/claude-work` など)の直下に、`CLAUDE.md` という名前でファイルを作成します。ここでは、セキュリティ診断報告書の作成を例に、シンプルな指示書を書いてみましょう。

```markdown
# 業務ルール

## 必ず守ること
- 報告書の中で顧客名は「お客様」に統一する
- 深刻度は「緊急/高/中/低」の4段階で表記する
- 報告書のドラフトを外部に送る前に、必ず自分が最終確認する

## できれば守ってほしいこと
- 文体は「です・ます調」で統一する
- 専門用語には簡単な補足説明を添える
```

ポイントは、抽象的な理想論ではなく、Claude Codeが読んだだけで実行できる具体的なルールを書くことです。「わかりやすく書いて」ではなく「専門用語には補足説明を添える」のように、判断基準がはっきりわかる表現を選びます。

### 手順2: CLAUDE.mdが読み込まれていることを確認する

作業フォルダで `claude` を起動し、次のように尋ねてみます。

```
このプロジェクトのCLAUDE.mdに書かれているルールを教えてください。
```

先ほど書いた内容がそのまま返ってくれば、CLAUDE.mdが正しく読み込まれています。続けて、実際に報告書の下書きを1つ依頼し、指示したルール(「お客様」表記や深刻度の4段階表記)が反映されているかを確認してみましょう。

## まとめ+チーム資産化アクション(5分)

- CLAUDE.mdは、毎回説明し直す必要をなくす"常時読み込まれる指示書"である
- スコープ(Managed/User/Project/Local)を理解し、まずはProjectスコープから始める
- Auto Memoryは、人間が書くCLAUDE.mdとは別に、Claude Code自身が学んだ経験則を蓄積する仕組み

**チームへの持ち帰りアクション**: 今日作ったCLAUDE.mdをチームメンバーと共有し、「他のメンバーならどんなルールを追加したいか」を出し合ってみてください。個人の気づきをCLAUDE.mdというファイルに落とし込むことで、初めてチームの資産になります。

## ベストプラクティスのひとこと

Anthropic公式のベストプラクティスでは、効果的なCLAUDE.mdについて次のような指針が示されています(参考: https://code.claude.com/docs/en/best-practices.md )。

- **1ファイルあたり200行以下を目安にする**。長くなりすぎたら、内容ごとにファイルを分けて@importする
- **Claude Codeが読んだだけでは推測できないことだけを書く**。「わかりやすい文章を書く」といった一般論ではなく、「深刻度は緊急/高/中/低の4段階で表記する」のように、そのプロジェクト固有のルールに絞る

筆者自身も、日常的に使っているCLAUDE.mdを「MUST(絶対に守ること)」「SHOULD(推奨されること)」「MAY(任意)」という3段階の見出しで分類しています。この分類を使うと、「これは絶対に外せないルールなのか、それとも好みの範囲なのか」がClaude Codeにも人間にも一目でわかるようになり、ルールが増えてきたときの見通しがよくなります。今日のハンズオンで作った「必ず守ること」「できれば守ってほしいこと」という2段構成も、この考え方を簡略化したものです。慣れてきたら、ぜひMUST/SHOULD/MAYの3段階に発展させてみてください。
