## 今日のゴール

Claude Codeが選べる複数のモデルと「effort(推論の深さ)」の違いを理解し、タスクに応じて使い分けられるようになる。

## 座学

### なぜモデルを使い分けるのか

これまでの回では、Claude Codeを「決まったモデルで動くツール」として使ってきました。しかし実際には、Claude Codeの中では複数のモデルを切り替えて使うことができます。モデルにはそれぞれ「思考力の深さ」と「応答速度・コスト」のトレードオフがあり、タスクの性質に応じて適切なモデルを選ぶことで、時間と利用枠の両方を効率的に使えるようになります。

Claudeのモデルには、大きく分けて次の3系統があります。

- **Opus系**: 最も思考力が高く、複雑な設計判断や曖昧な要件の解釈に強い。応答生成に時間がかかり、利用できる回数の消費(レート制限)も大きい
- **Sonnet系**: 思考力と速度のバランスが良く、指示が明確になっているタスクの実行に向く。日常的な作業の主力になるモデル
- **Haiku系**: 応答が非常に速く、軽量なタスクや大量の定型処理に向く。複雑な判断はOpus・Sonnetに比べて不得手

セキュリティコンサルティングの業務に例えると、Opusは「調査方針を練るシニアコンサルタント」、Sonnetは「方針に沿って手を動かすアナリスト」、Haikuは「定型チェックを高速にこなすアシスタント」のようなイメージで捉えると分かりやすくなります。

### `/model` コマンドでモデルを切り替える

Claude Codeのセッション内で `/model` と入力すると、現在使用しているモデルの確認や切り替えができます。プロジェクトの `settings.json` にモデルを固定で指定しておくことも可能です。

```json
{
  "model": "claude-sonnet-5"
}
```

### 推奨構成: Plan Modeと実装フェーズでモデルを使い分ける「opusplan」

第2回で扱ったPlan Mode(実際に変更を加える前に、まず計画だけを立てるモード)を思い出してください。Plan Modeで行う「何を、どの順番で進めるか」という計画立案は、曖昧な要件を整理し、抜け漏れのない手順に落とし込む必要がある、思考力が問われる作業です。一方、計画が固まったあとの実装フェーズは、決まった手順を着実にこなしていく作業が中心になります。

そこで有効なのが、Plan Mode時にはOpus系の高性能モデルを使い、実際の実装フェーズに入ったらSonnet系に切り替えるという設定です。Claude Codeにはこの使い分けを自動化する `opusplan` という設定があり、Plan Modeでは思考力の高いモデルを、それ以外の作業では高速なモデルを自動的に使うようになります。

```json
{
  "model": "opusplan"
}
```

「計画は高い思考力のモデルで慎重に、実行は高速なモデルでテンポよく」という考え方は、モデル選択に限らず、人間のチームでの役割分担にも通じる発想です。この構成は日常的な運用における推奨設定の一つとして覚えておいてください。

### `/effort` で推論の深さを調整する

Claude Codeには、モデルそのものの選択とは別に、「effort(推論の深さ)」という調整軸があります。`/effort` コマンドで、以下のようなレベルを選択できます。

- `low`: 素早く簡潔に応答する。単純な質問や軽い作業向け
- `medium`: 標準的なバランス
- `high`: じっくりと検討してから応答する。複雑な判断や、間違いが許されない作業向け
- `xhigh` / `max`: さらに深く考え込む。非常に複雑な設計や、リスクの高い判断が必要な場面向け

同じモデルであっても、effortを上げることで、より慎重に、多角的に検討した上で応答するようになります。Anthropic公式のブログ記事(https://claude.com/blog/claude-model-and-effort-level-in-claude-code )によれば、effortは単に「考える時間の長さ」を調整するものではなく、読み込むファイルの数、検証の徹底度、マルチステップタスクで進捗確認を挟むまでの距離など、Claudeが実行する作業量全体を制御するパラメータだと説明されています。

同記事が推奨する運用は、タスクごとに細かく手動で切り替えるのではなく、effortを「自分の作業スタイル全体に対する基本設定」として扱うことです。多くのタスクではモデルの既定のeffortレベルのままで十分であり、Claudeの応答が期待に届かなかったときに、初めて次のように原因を切り分けて調整先を判断します。

- **知識・判断力が足りていない**(的外れな方針、専門知識の誤り)と感じる場合 → より上位のモデルに切り替える
- **検討や確認の「手数」が足りていない**(ファイルを読み飛ばした、テストを実行しなかった)と感じる場合 → effortレベルを上げる

つまりモデルは「どれだけの能力を持っているか」を決め、effortは「その能力を使ってどこまで丁寧にやり切るか」を決める、という役割分担になります。セキュリティ診断の設計や複雑な脅威分析など、判断ミスの影響が大きい業務ではこの切り分けを意識して `xhigh` や `max` まで上げることも検討してください。

### 日本語利用時の注意点: トークン消費とコスト

Claudeは入力・出力された文字量に応じて「トークン」という単位で処理量を消費します。日本語は英語に比べて同じ内容を表現するのに必要なトークン数が多くなる傾向があり、目安として英語の1.5倍前後のトークンを消費するといわれています。これは、日本語でのやり取りが英語に比べてレート制限(一定時間内に使える利用枠)に早く到達しやすいことを意味します。

長い会議メモや調査資料をそのまま貼り付けるのではなく、必要な部分だけを渡す、要点を絞って質問するといった工夫は、コスト面でも有効です。この点は第2回で扱ったコンテキスト管理の考え方ともつながっています。現在の利用状況は `/usage`(エイリアス: `/cost`)コマンドで確認でき、セッションごとのコストやプランの利用上限に対する消費状況が表示されます。レート制限が気になったら、まずこのコマンドで現状を把握する習慣をつけましょう。

### 利用プランとレート制限

Claude Pro / Max / Team といった利用プランによって、一定時間内に利用できるモデルの種類や量(レート制限)が異なります。上位プランほど、Opus系モデルを含めてより多くの利用枠が用意されています。コンサルティング業務でClaude Codeを使い倒す場合、Opus系モデルを多用する調査・計画立案の作業が多い方ほど、上位プランの恩恵を受けやすくなります。自分の業務内容と利用頻度を踏まえて、チームでプランを検討してみてください。

### `advisor` コマンド: 必要な場面だけ上位モデルを呼び出す

ここまでは「セッション全体でどのモデルを使うか」という話でしたが、Claude Codeには、普段は軽量なモデルで作業しつつ、判断に自信が持てない局面だけピンポイントで上位モデル(Fable 5など)にレビューを依頼できる `advisor` というスラッシュコマンドが用意されています。

`advisor` を実行すると、これまでの会話履歴やツール呼び出しの結果がまるごと上位モデルに渡され、今の進め方が妥当かどうかのセカンドオピニオンを得られます。実装の方針を決める前、行き詰まったとき、あるいは非自明なタスクを完了したと判断する前に呼び出すのが典型的な使い方です。

この仕組みが効率的なのは、セッション全体を高性能モデルに固定するのではなく、「判断の要所」だけに絞って呼び出す点にあります。普段の実行はSonnetやHaikuなどのコストの低いモデルに任せ、方針の妥当性確認のような思考力が問われる瞬間だけFable 5のような上位モデルを使うことで、全体のコストを抑えながら重要な判断の質を落とさずに済みます。これは、本節冒頭で述べた「Opusは調査方針を練るシニアコンサルタント」という例えを、セッション単位ではなく1回のレビュー単位で実現するイメージです。

**注意点**: `advisor` はclaude.aiのサブスクリプションプラン(Pro / Max / Team)で利用できる機能であり、Amazon Bedrock経由でClaude Codeを利用している環境では使用できません。社内でBedrock経由の構成を採用している場合は、この点を踏まえて利用可否を確認してください。

## ミニハンズオン

同じタスクを、モデルとeffortを変えながら試してみましょう。

### 手順1: モデルを確認する

```
/model
```

現在のモデル設定が表示されます。

### 手順2: あるタスクをSonnet(標準構成)で試す

手元にある調査メモや報告書が3件ほどあれば、それらを作業フォルダに置いてみましょう(架空の内容で構いません)。準備ができたら、少し込み入った依頼をしてみます。

```
このフォルダにある3つの調査メモを比較して、共通するリスクと相違点を整理してください。
```

### 手順3: 同じタスクをeffortを上げて試す

```
/effort high
```

と設定してから、同じ依頼をもう一度出してみます。応答までにかかる時間や、整理の粒度・観点の深さの違いを比較してみてください。

### 手順4: Plan Modeでの挙動を確認する

`opusplan` が設定されている場合、Plan Modeに入った瞬間にモデルが自動的に切り替わります。Shift+Tabなどでplan modeに入り、`/model` を再度実行して、実際にモデルが変わっていることを確認してみましょう。

## まとめ+チーム資産化アクション(5分)

- モデルには「思考力」と「速度・コスト」のトレードオフがあり、Opus・Sonnet・Haikuで役割が異なる
- 計画立案は高性能モデル、実装は高速モデルという使い分け(`opusplan`)が推奨構成
- `/effort` でタスクの重要度に応じて推論の深さを調整できる
- `advisor` コマンドを使えば、普段は軽量モデルで作業しつつ判断の要所だけ上位モデルのセカンドオピニオンを得られ、コストと判断品質を両立できる
- 日本語はトークン消費が多くなりやすい点を踏まえ、渡す情報は要点を絞る

**チームへの持ち帰りアクション**: 自分がよく行う業務(調査、報告書作成、診断補助など)を思い浮かべ、「これは計画寄りの作業か、実行寄りの作業か」を意識的に分類してみてください。次回はその分類をもとに、業務手順そのものをスキルとして切り出していきます。なお、モデルの使い分けを仕組みに組み込みたい場合は、第9回で扱うカスタムサブエージェント(agent定義ファイルの中でモデルを指定できます)が対応する機能になるので、覚えておいてください。

## ベストプラクティスのひとこと

Anthropic公式のベストプラクティスでは、「探索(Exploration)→計画(Planning)→実装(Implementation)→検証(Verification)」という4つの段階でタスクを進めることが推奨されています(参考: https://code.claude.com/docs/en/best-practices.md )。この4段階のうち、探索と計画の段階は曖昧な情報を整理し方針を定める必要があるため思考力の高いモデルが力を発揮しやすく、実装と検証の段階は決まった手順を確実にこなす作業が中心になるため高速なモデルとの相性が良いとされています。今日紹介した「opusplan」という設定は、まさにこの考え方をそのまま体現したものです。
