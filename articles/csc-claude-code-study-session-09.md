## 今日のゴール

「もう一人の自分」に調査や作業を任せる仕組みであるサブエージェントの考え方を理解し、実際に使ってみる。

## 座学

### サブエージェントとは何か

第2回で、Claude Codeにはコンテキストウィンドウ(会話の文脈を保持できる容量)という限りある資源があり、これを使い切ると挙動が不安定になることを説明しました。

https://zenn.dev/cscloud_blog/articles/csc-claude-code-study-session-02

調査や下調べのために大量のファイルを読ませたり、Web検索を何度も行わせたりすると、そのやり取り自体がコンテキストを圧迫してしまいます。

この問題を解決するのが「サブエージェント」です。サブエージェントとは、メインの会話とは別に、独立したコンテキストを持つ「もう一人の自分」を呼び出し、そこに調査や作業を委譲する仕組みです。サブエージェントが調べた大量の情報はサブエージェント側のコンテキストで処理され、メインの会話には要点だけがまとめられて返ってきます。これにより、メインの会話のコンテキストを汚さずに、重い調査を進めることができます。

### 組み込みのサブエージェント

Claude Codeにはあらかじめ用意されたサブエージェントがいくつかあります。代表的なものは次のとおりです。

| サブエージェント | 役割 |
|---|---|
| Explore | 読み取り専用でコードやファイルを高速に探索する。「このキーワードはどこで使われているか」といった調査向け |
| Plan | 実装や作業の計画を設計する。複数の選択肢を比較検討させたいときに使う |
| general-purpose | 特定の役割に限定されない汎用的な調査・作業タスクに使う |

これらは特別な設定をしなくても、Claude Code自身が「今回は調査を委譲した方が良さそうだ」と判断した場合に自動的に呼び出したり、こちらから明示的に指名して呼び出したりできます。

### カスタムサブエージェントを作る

組み込みのサブエージェントに加えて、`.claude/agents/` というディレクトリに専用の定義ファイルを置くことで、自分たちの業務に特化したサブエージェントを作ることができます。たとえば「セキュリティコードレビュー専門」「依存ライブラリの脆弱性監査専門」といったように、役割を絞り込んだ専門agentを用意しておくと、その都度長い指示を書かなくても、名前を呼ぶだけで専門的な視点でのレビューを任せられるようになります。

筆者の環境では、OWASP Secure Agent Playbookというプラグイン(プラグインについては前回扱いました)を導入しており、その中には次のような専門特化型のサブエージェントが含まれています。

- コードのセキュリティ脆弱性だけをレビューする専門agent
- APIの設計をOWASP API Security Top 10の観点でレビューする専門agent
- 依存ライブラリの脆弱性(CVE)を監査する専門agent
- これら複数の専門agentを束ね、並列に走らせて結果を統合する「security-team-lead」のような統括役agent

統括役agentに「このプロジェクトを総合的にセキュリティレビューして」と依頼すると、内部で複数の専門agentが同時並行で異なる観点からレビューを行い、最後にその結果がまとめて報告されます。1つのagentに何もかも任せるより、観点ごとに専門agentへ分業させた方が、見落としが少なく、レビューにかかる時間も短縮できます。

### Agent Team(実験的機能)

さらに発展的な仕組みとして、複数のサブエージェントが単発の作業委譲にとどまらず、チームとして継続的に協調しながら動く「Agent Team」という機能があります。これは執筆時点では実験的機能であり、今後仕様が変更される可能性がある点に注意してください。筆者の環境では、設定ファイルに `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` という環境変数を追加することで有効化しています。実験的機能を業務で使う際は、まず個人の検証環境で試し、挙動が安定していることを確認してから展開するようにしてください。

### TAM業務での並列化ユースケース

サブエージェントによる並列化は、コーディング作業に限った話ではありません。TAM業務でも、たとえば次のような場面で活用できます。

- 1つの提案書や設計書に対して、複数の異なる観点(セキュリティ、コスト、運用性など)から同時にレビューを走らせる
- 複数の情報源(公式ドキュメント、社内資料、過去の類似案件)を並行して調査し、後でまとめて比較する
- 診断対象のシステムについて、観点の異なる複数の専門agentに一次チェックを分担させる

## ミニハンズオン

### 手順1: 調査をサブエージェントに任せてみる

作業フォルダの中に、複数の資料ファイルを用意します。過去に作成した報告書や調査メモなど、手元にあるテキストファイルを3件ほどコピーしておくとよいでしょう。手元に適当なファイルがない場合は、Claude Code自身に「架空のセキュリティ診断調査メモを3つ作って」のように頼んでサンプルを用意してもらっても構いません。準備ができたら、次のように依頼します。

```
このフォルダにあるファイルを調査して、共通して出てくるキーワードを
探してほしい。調査はサブエージェントに任せて、結果だけ報告して。
```

Claude Codeが調査の委譲を判断すると、Exploreやgeneral-purposeといったサブエージェントが呼び出され、ファイルの読み込みや探索が行われます。手元の会話画面には、その過程の詳細ではなく、要点をまとめた報告だけが返ってくることを確認してください。

### 手順2: 観点を分けて並列レビューを試す

同じ資料に対して、次のように依頼してみます。

```
この文書を、セキュリティの観点とコストの観点それぞれから
レビューしてほしい。2つの観点は別々のサブエージェントに
分担させて、それぞれの指摘を並べて報告して。
```

1つのagentがすべての観点を順番にこなすのではなく、観点ごとに担当が分かれて並行して動くことで、待ち時間を短縮しながら多角的な指摘を得られることを体感してください。

## まとめ+チーム資産化アクション(5分)

- サブエージェントは、メインの会話のコンテキストを守りながら重い調査や作業を任せるための仕組みである
- 組み込みのExplore・Plan・general-purposeに加え、`.claude/agents/` で業務特化の専門agentを自作できる
- 観点ごとに専門agentへ分業させることで、レビューの見落としを減らし、時間も短縮できる
- Agent Teamは実験的機能であり、業務利用の前に個人環境での検証を挟むこと

**チームへの持ち帰りアクション**: 自分たちの業務でよく発生する「複数の観点からのチェック」を1つ洗い出し、それぞれの観点をサブエージェントに分担させる依頼文を作ってみてください。うまくいった依頼文は、第6回で扱ったスキル化の候補としてメモしておくと、チーム資産として蓄積できます。

## ベストプラクティスのひとこと

Anthropic公式のベストプラクティスでは、「調査はサブエージェントに委譲し、メインの会話のコンテキストを温存する」ことが推奨されています(参考: https://code.claude.com/docs/en/best-practices.md )。あわせて、会話が長くなるとパフォーマンスが低下する兆候にも注意が必要です。

| 兆候 | 考えられる原因 | 対処法 |
|---|---|---|
| 同じ間違いを繰り返す | コンテキストが埋まり、以前の指示が埋もれている | `/compact` で会話を要約する |
| 以前の決定と明らかに矛盾する回答をする | CLAUDE.mdや過去の指示が読み込まれていない | `/memory` で内容を確認し、記述を簡潔にする |
| 修正を依頼しても同じ箇所に戻ってくる | 計画を立てずにいきなり実装や作業を進めている | `/plan` で改めて計画段階に戻る |

こうした兆候が見えたら、無理に会話を続けるのではなく、一度立ち止まって `/compact` や `/clear`、あるいは `/plan` への立ち戻りを試すようにしましょう。
