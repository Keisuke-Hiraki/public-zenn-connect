## なぜブログを書くのか？

世の中にはブログと呼ばれるものは多々ありますが今回は技術ブログに限定したいと思います。

個人的にブログを書く理由としては以下です。

- 顧客へ所定の手順や情報を共有する際に 1 つブログを書けば複数の顧客に展開でき工数削減になるから
- アウトプットに必要なインプットができるから
- アウトプットする習慣を身に付けることで対外的なアピールが容易になるから

### 工数削減

複数の顧客を持ち、顧客に対してさまざまな技術に関する問い合わせや是正に関する案内をお仕事の中で実施する機会があります。
その中で同じような情報を共有することはしばしばあります。

その際に顧客毎に資料を作成して説明するより、一般化された情報に置き換えブログとして共有することで顧客毎に説明資料を作成することなくブログを用いて説明することができます。

また、そのブログを社内に共有することで他の社員の工数も削減できチームとしての効率を上げることも可能です。

### インプット

ブログに限らず誰かに何かを伝えるという行為には、想像しているよりも多くの知識やその情報に対する理解が必要になります。

自分の頭の中では分かっていても、他人に説明するとなるとなかなか説明できないという感じです。

ブログや LT ・登壇など、アウトプットするためには想像以上のインプットを行うので一つの案件をこなしたくらいの学習に繋げられます。

### アピール

自分自身はプログラマーではないのでなかなか世間に自分の実力を証明する方法としてポートフォリオとしてアプリケーションを作るみたいなことはなかなかできません。（生成 AI 使えばできますが…）

インフラ・クラウドエンジニアの類は自己アピールする材料として自分の経験を相手に伝える重みがかなり変わってきます。

ブログを公開することにより、より対外的にどんな知識を有していて、どんな人なのかということをアピールするものとしてかなり優秀なものになります。

## ブログの書き方

ブログを書くことの良さを知っていただいたところでブログの内容も肝心ですが、ブログ公開にあたって知っておくべきことについて記載します。


### 公開するプラットフォーム

ブログを公開するプラットフォームは様々あります。

- Zenn
- Qiita
- note

CSC では現在、Zenn の Publication を活用しているため Zenn を活用することを前提としますが、
プラットフォームによって特色が異なるのでぜひいろいろ触って試してみてください。

### マークダウン記法

マークダウンは、文章の構造や装飾を特定の記号を使用して表現するマークアップ言語の一つで、HTML や XML などの仲間です。

ブログを記述する際は原則マークダウン（Markdown）という表記方法を活用します。

マークダウンを活用することで例えばこのようなのっぺりとした文章が、

```md
レポート：新製品開発計画の概要

１．はじめに
市場のニーズと競合他社の動向を分析した結果、高機能かつ手頃な価格帯の新製品を投入する方針を決定しました。

２．製品の主要機能
高性能プロセッサの搭載
特に、処理速度を現行モデルの二倍に向上させることが目標です。
長時間バッテリーの実現
ユーザーからの要望が最も多かった点に対応します。

３．今後のスケジュール
今月中にプロトタイプの最終確認を完了し、来月上旬から量産体制に入ります。

重要：この計画は機密情報です。
```

以下のように見出しやリスト、リンクなどを簡単に装飾できます。

```md
# レポート：新製品開発計画の概要

## 1. はじめに
市場のニーズと競合他社の動向を分析した結果、高機能かつ手頃な価格帯の新製品を投入する方針を決定しました。

## 2. 製品の主要機能
- **高性能プロセッサの搭載**
    - 特に、処理速度を現行モデルの二倍に向上させることが目標です。
- **長時間バッテリーの実現**
    - ユーザーからの要望が最も多かった点に対応します。

## 3. 今後のスケジュール
今月中にプロトタイプの最終確認を完了し、来月上旬から量産体制に入ります。

> **重要:** この計画は機密情報です。
```

主要な記法を紹介します。

#### 見出し

まずは文章の構成を分け、読み手にどんなことを今書いているかを明確に項目として分ける、**見出し**です。

見出しは、`#`の個数によって見出しのレベルを設定できます。

```md
# 見出しレベル 1（ブログ記事のタイトルに使用されるため、本文では使用しない）

## 見出しレベル 2（章立てに使用）

### 見出しレベル 3（サブセクションに使用）
```

:::message
基本的にブログの文章で使用する見出しは見出しレベル`2`から設定することが多いので覚えておきましょう。
:::

#### リスト

箇条書きや名前の一覧を記載したい場合に使用されるのが、**リスト**です。

通常のリストと番号付きのリストがあるのでユースケースに応じて使い分けましょう。

字下げをしたい場合は、インデントを使用しズラすことで字下げが可能です。

```md
- 親項目 1
    - 子項目 1
    - 子項目 2
- 親項目 2

1. 番号付き項目 1
2. 番号付き項目 2
```

- 親項目 1
    - 子項目 1
    - 子項目 2
- 親項目 2

1. 番号付き項目 1
2. 番号付き項目 2

#### テーブル

表形式で情報を整理したい場合に使用されるのが、**テーブル**です。

テーブルを活用することで、事柄の比較や説明がごちゃごちゃせず、綺麗に見えます。

```md
| 項目 | 説明 |
| --- | ---|
| 項目 1 | 項目 1 の説明 |
| 項目 2 | 項目 2 の説明 |
```

| 項目 | 説明 |
| --- | ---|
| 項目 1 | 項目 1 の説明 |
| 項目 2 | 項目 2 の説明 |

#### そのほか

その他さまざまありますが、プラットフォームによって使える記法などが微妙に変わったりします。

使用する際はどんなの記法が使えて何が使えてないか、プレビュー画面で必ず確認し、表記が崩れてないか確認してから公開しましょう。

### ブログのネタ

ブログを書く際に一番悩むのがブログのネタです。

何を書けばいいのか、自分なんかの知識をブログに公開してもいいんだろうか、と考えた結果書かない選択肢を取る人は少なくないと思います。

ただ一つ言えることとしては、

「アウトプットしていることがえらい！」

です。

自分の持っている知識を自分だけに閉ざしていることは、内容によっては希少性といった観点で価値があるかもしれないです。

ただ組織や業界全体からしたらメリットはないといっても過言ではないと思います。

なので、まずは自分が知っていること、経験したことをアウトプットすることに価値があると理解しましょう。  
どんな些細なことでも大丈夫なので気付きを得られたらどんどんアウトプットしていくことが大事です。

文章にするのが難しいようであれば Zenn にはスクラップという機能もあり、メモとして書き留めておくこともできます。
これは Twitter のような感覚で技術的な気づきをメモできる機能で、以下のような使い方ができます：

- 学習ログとして記録
- トラブルシューティングの備忘録
- 後でブログにまとめるためのネタ帳

まずはこういったところからチャレンジするのも良いかもですね。

https://zenn.dev/zenn/articles/about-zenn-scraps

:::message alert
当然アウトプットすればどんな内容でもえらいのかというともちろんそうではないケースもあります。
他者・プロダクトを非難するような内容や公序良俗に反する内容などプラットフォーム毎に規約やガイドラインとしてどんなものを公開してはいけないか・避けるべきかを明文化していたりします。
ブログを公開する前に必ず確認し、違反しないように注意しましょう。
