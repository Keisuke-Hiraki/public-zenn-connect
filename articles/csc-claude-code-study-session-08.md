## 座学

### MCPとは何か

MCP(Model Context Protocol)は、AIエージェントと外部のツールやサービスを接続するためのオープンな標準規格です。これまでの回で扱ってきたSkillsやCLAUDE.mdが「Claude Codeに指示や手順を教える仕組み」だったのに対し、MCPは「Claude Codeが外部の情報源やシステムに直接アクセスできるようにする仕組み」だと捉えると分かりやすいでしょう。

例えば、Notion上のドキュメントを検索したい、AWS環境の設定をスキャンしたい、ブラウザで実際の画面を確認したい、といった要望は、Claude Code単体(ファイルの読み書きとコマンド実行のみ)では実現できません。MCPサーバーを接続することで、こうした外部連携が可能になります。

### MCPサーバーのスコープと接続方法

MCPサーバーの設定は、どの範囲で有効にするかによって3つのスコープに分かれます。

| スコープ | 保存先 | 有効範囲 |
|---|---|---|
| local | 個人のローカル設定 | 自分が使う特定のプロジェクトのみ |
| project | プロジェクト内の `.mcp.json` | Gitで共有され、チームメンバー全員が利用可能 |
| user | 個人のユーザー設定 | 自分が使うすべてのプロジェクト |

接続には `claude mcp add` コマンドを使います。代表的な接続方法は2種類です。

:::message
`claude mcp add` コマンドは、macOSのターミナルでもWindowsのPowerShellでも同じコマンドをそのまま実行できます。
:::

**HTTP接続(リモートサービスに多い形式)**
```bash
claude mcp add --transport http notion https://mcp.notion.com/mcp
```

**ローカルコマンド接続(手元で動くツールを呼び出す形式)**
```bash
claude mcp add --transport stdio playwright -- npx -y @playwright/mcp
```

接続後は、Claude Codeのセッション内で `/mcp` コマンドを実行すると、接続済みのサーバー一覧や認証状態を確認できます。初回接続時にログインを求められるサービス(NotionやSlackなど)もあります。

### 診断業務で役立つMCPサーバーの例

以下は、実際にセキュリティ関連の業務で活用しやすいMCPサーバーの例です。ここで挙げるのは一例であり、自社の業務内容に応じて必要なものを選んで導入してください。

- **context7**: ライブラリやフレームワークの最新の公式ドキュメントを参照するためのMCP。診断対象のシステムが使っているフレームワークの仕様を、学習データの古さに左右されずに確認したいときに役立つ
- **playwright**: ブラウザを実際に操作して画面のスクリーンショットや表示内容を確認するためのMCP。Webアプリケーションの挙動を目視で確認しながら調査を進めたいときに使う
- **notion**: Notion上のドキュメントを検索・作成するためのMCP。過去の診断報告書や社内ナレッジを参照したり、調査結果をNotionページとしてまとめたりする際に使う
- **AWSセキュリティ診断系のMCP**: AWS上のコードやインフラ構成に対してセキュリティスキャンを実行するためのMCP。クラウド環境の設定不備やコードの脆弱性を、対話しながら調査できる

これらは「調査対象についての最新情報を得る」「実際の画面を確認する」「過去の知見を参照する」「専門的なスキャンを実行する」という、診断業務の異なる場面にそれぞれ対応しています。

### 組み込みのセキュリティレビュー機能

Claude Codeには `/security-review` という組み込みコマンドがあり、現在作業しているブランチの変更内容に対してセキュリティ上のリスクを検査できます。MCPで外部の専門スキャナーを呼び出す方法と、こうした組み込み機能を使い分けることで、状況に応じた調査の幅が広がります。

### 調査からレポート作成までを一気通貫で任せる

MCPを組み合わせることで、次のような一連の流れをClaude Codeに任せられるようになります。

1. 調査対象システムの情報を、ドキュメントやコードから収集する
2. 収集した情報をもとに、リスクや懸念点を分析する
3. 分析結果を、報告書のフォーマットに沿ってドラフト化する

例えば「このリポジトリのAPIエンドポイントを一覧化し、認証の有無を確認したうえで、リスクの高い順に並べたレポートのドラフトをNotionに作成してください」といった指示は、複数のMCP(コード読み取り、分析、Notionへの書き込み)を横断する一つの作業として、Claude Codeがまとめて実行できます。

### AIによる一次診断と、人間による最終判断

MCPやセキュリティレビュー機能を使えば、多くの作業を自動化できますが、それがそのまま「人間の確認が不要」を意味するわけではありません。

mixi社が公開したブログ「Claude Codeにセキュリティ診断をさせてみた」(https://zenn.dev/mixi/articles/79831816f9fd22 )では、「経験豊富なセキュリティ専門家として」というロールプロンプトを与えたうえで自社アプリケーションを診断させたところ、SQLインジェクション(重大度: High)を含む5件の脆弱性を検出し、GitHub上のIssue化から修正までを実施できたと報告されています。一方で同記事は、大規模なシステムや、そのシステム固有のビジネスロジックに根差した脆弱性の判断には、依然として人間の専門知識が必須であるという限界も正直に述べています。

このことから導かれる実務上の心得はシンプルです。**AIによる診断は「一次診断」として活用し、最終的なリスク判断と顧客への報告は、必ず人間のコンサルタントが行う。** これは効率化のための自動化であって、専門家としての判断責任を手放すものではない、という点を今回のシリーズを通じて改めて強調しておきます。

## ミニハンズオン

### 手順1: MCPサーバーを1つ接続する

すでに公式ドキュメントを参照しながら、自分の業務に関係のありそうなMCPサーバーを1つ選び、接続してみましょう。ここでは例としてcontext7を接続します。このコマンドはmacOSのターミナルでもWindowsのPowerShellでも同じように実行できます。

```bash
claude mcp add --transport http context7 https://mcp.context7.com/mcp
```

### 手順2: 接続を確認する

Claude Codeのセッション内で以下を実行し、正しく接続されているか確認します。

```
/mcp
```

一覧に接続したサーバーが表示され、ステータスが有効になっていれば準備完了です。

### 手順3: 実際に調査を依頼してみる

```
（自分が普段使っているフレームワークやライブラリ名)の最新のセキュリティに関する推奨設定を調べて、3点にまとめてください。調べた内容の根拠となる情報源も明記してください。
```

「根拠となる情報源も明記してください」のように、検証の手がかりを指示に含めておくと、回答の裏付けを自分で確認しやすくなります。

### 使わないMCPサーバーはつなぎっぱなしにしない

MCPサーバーを増やすと便利な半面、「常時接続しているサーバーが多いほどコンテキスト(会話の文脈情報の容量)を圧迫するのでは」と気になる方もいるかもしれません。Claude Codeには「MCP tool search」という仕組みが既定で有効になっており、各ツールの詳細な定義(スキーマ)はセッション開始時には読み込まれず、実際にそのツールが必要になった場面で初めて読み込まれます(参考: https://code.claude.com/docs/en/mcp.md )。この仕組みにより、接続するMCPサーバーの数を増やしても、コンテキストへの影響は最小限に抑えられます。

とはいえ、接続中の各サーバーの「ツール名」と「サーバーからの利用案内(server instructions)」自体は、セッション開始のたびに読み込まれます。そのため、今の案件では使っていないサーバーを接続したままにしておくと、わずかとはいえ無駄なコンテキスト消費が積み重なっていきます。使わなくなったサーバーは `claude mcp remove <サーバー名>` で削除する、あるいはプロジェクトの `.claude/settings.json` で `enabledMcpjsonServers` / `disabledMcpjsonServers` を使って案件ごとに有効なサーバーを絞り込む、といった整理を習慣にしておくとよいでしょう。

## まとめ+チーム資産化アクション(5分)

- MCPはClaude Codeと外部のツール・サービスを繋ぐ標準規格であり、local/project/userのスコープで管理する
- 調査・分析・レポート作成という一連の流れを、複数のMCPを横断してClaude Codeに任せられる
- tool search機能によりコンテキストへの影響は最小限に抑えられているが、使わないサーバーは`claude mcp remove`などで整理する習慣をつけるとよい
- AIによる診断はあくまで一次診断であり、最終判断は人間のコンサルタントの責任である

**チームへの持ち帰りアクション**: 今日接続したMCPサーバーの設定を、プロジェクトスコープ(`.mcp.json`)でチームに共有し、チーム全体で同じ調査環境を再現できるようにしてください。

## ベストプラクティスのひとこと

Anthropic公式のベストプラクティスでは、「Claudeに検証の手段を与える」ことの重要性が説かれています(参考: https://code.claude.com/docs/en/best-practices.md )。「〜を調べてください」という指示だけでなく、「その根拠となる情報源も示してください」「期待する形式はこうです」といった、結果を自分で確認できるための手がかりを添えることで、Claude Codeは自己検証を挟みながら精度の高い作業を進められます。今日のハンズオンで加えた「根拠となる情報源も明記してください」という一文も、この原則の実践例です。
